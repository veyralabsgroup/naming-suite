## Example 2: Developer CLI Tool

**User input:** "Building a CLI tool for managing environment variables across projects"

---

## DomainForge Analysis — Environment Variable CLI Tool

**Archetype detected:** DevTool / CLI
**Naming mode:** Indie Hacker + DevTool (short, technical, lowercase-first)

---

### Top Recommendations

#### 1. envx.dev — 94/100
`env` (universally understood in developer context) + `x` (the execution/extension convention). It reads naturally in a terminal: `npx envx init`, `envx push`, `envx pull`. The `.dev` TLD makes it unambiguous. Extremely memorable in the developer community.

**Domain:** envx.dev — check availability (~$12/yr)
**Social:** @envxdev on GitHub (primary), @envx on X
**Trademark:** Clean

#### 2. dotforge.dev — 88/100
.dotfile + forge. Developer community understands `.` prefix conventions immediately. Strong combination of familiar reference and creation energy.

**Domain:** dotforge.dev — available (~$12/yr)
**Social:** @dotforge — likely available
**Trademark:** Clean

#### 3. vaultx.sh — 84/100
Vault (secure storage) + x (CLI convention). `.sh` TLD is a cultural signal for shell tools. `vault` communicates security without overthinking it.

**Domain:** vaultx.sh — check availability (~$10/yr)
**Social:** @vaultx — moderate availability
**Trademark:** Flag — HashiCorp Vault is well-known. Consider alternative if going commercial.

---

### Also Strong

- **dotenv.run** — 80/100 — `dotenv` is the exact term of art, .run implies CLI execution. Very keyword-rich.
- **envault.dev** — 78/100 — `env` + `vault`. Descriptive but clever portmanteau.
- **quark.sh** — 76/100 — Physics reference (smallest unit). Clean, 5 chars, .sh TLD. Perfectly brandable.
